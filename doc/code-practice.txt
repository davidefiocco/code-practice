1. INTRODUCTION                                        *code-practice-intro*

Code Practice is a Neovim plugin for browsing coding exercises, solving them,
and running tests -- all without leaving the editor.

Features:

- Floating browser UI with split preview pane
- Deterministic navigation: next, skip, previous with session history
- Extensible engine registry: interpreted and compiled runners
- Theory (multiple-choice) questions with answer checking
- Results window with per-test pass/fail detail
- Reference solution viewer
- Solved/unsolved status indicators in the browser
- AI-powered contextual hints (open-ended and multiple-choice aware)
- LLM-powered exercise generation from a configurable syllabus

2. INSTALLATION                                      *code-practice-install*

Using lazy.nvim: >lua

    {
      "davidefiocco/code-practice",
      dependencies = {
        "MunifTanjim/nui.nvim",
        "kkharji/sqlite.lua",
      },
      config = function()
        require("code-practice").setup()
      end,
    }
<
Requirements:

- Neovim 0.10+
- MunifTanjim/nui.nvim (floating window UI)
- kkharji/sqlite.lua (database persistence)
- Engine executables for each enabled engine (see |code-practice-engines|)

Run |:checkhealth| |code-practice| to verify all dependencies.

After installing, populate the exercise database with |:CP| `import` or
|:CP| `generate`.

3. QUICK START                                    *code-practice-quickstart*

1. Open the browser          |:CP|
2. Navigate with `j`/`k`     Press `Enter` to open an exercise
3. Write your solution        Edit the buffer
4. Run tests                  `Ctrl-t`
5. Move on                    `Ctrl-n`

4. COMMANDS                                        *code-practice-commands*

Everything goes through a single `:CP` command with subcommands.
Tab completion is supported: type `:CP <Tab>` to explore.

                                                                        *:CP*
:CP                     Open the exercise browser (same as `:CP open`).

:CP open                Open the exercise browser.

:CP close               Close the exercise browser.

:CP refresh             Refresh the exercise browser list.

:CP stats               Show practice statistics (total, solved, by
                        difficulty).

:CP help                Show an in-editor keymap cheat-sheet reflecting your
                        current configuration. For full documentation, see
                        this help file.

:CP import {path}       Import exercises from a JSON file. The expected
                        schema matches `test/example_exercises.json` in the
                        plugin source.

:CP import! {path}      Replace all existing exercises, test cases, attempts,
                        and theory options with the contents of {path}.

:CP generate            Interactively generate exercises via Hugging Face
                        LLM. Prompts for topic, count, difficulty, and
                        engine. See |code-practice-generate| for setup.

Exercise-level actions (run tests, next, skip, hints, solution, etc.) are
available via buffer-local keymaps only. See |code-practice-keymaps-exercise|.

5. KEYMAPS                                          *code-practice-keymaps*

All keymaps are configurable via |code-practice-config|. The defaults are
listed below.

BROWSER KEYMAPS ~
                                                 *code-practice-keymaps-browser*

Active inside the browser window opened by |:CP|.

    Key            Action ~
    `j` / `k`        Move selection down / up
    `<Down>` / `<Up>`  Move selection down / up
    `Enter` / `o`    Open selected exercise
    `e`              Toggle filter: easy difficulty
    `m`              Toggle filter: medium difficulty
    `h`              Toggle filter: hard difficulty
    `a`              Clear all filters
    `{filter_key}`   Toggle filter by engine (each engine defines its
                     own key in the registry; see |code-practice-engines|)
    `gg`             Go to top of list
    `G`              Go to bottom of list
    `q` / `Esc`      Close browser
    `?`              Show keymap cheat-sheet (|:CP| `help`)

EXERCISE BUFFER KEYMAPS ~
                                                *code-practice-keymaps-exercise*

Active inside exercise buffers (normal mode only). All exercise keymaps
use Ctrl shortcuts for single-chord access. Keymaps are configurable
via `keymaps.exercise` in config.

    Key            Action ~
    `Ctrl-t`         Run tests
    `Ctrl-n`         Next exercise
    `Ctrl-p`         Previous exercise
    `Ctrl-k`         Skip exercise
    `Ctrl-i`         Show hints
    `Ctrl-l`         View solution
    `Ctrl-d`         Show description
    `Ctrl-b`         Open browser

6. CONFIGURATION                                     *code-practice-config*

Pass a table to `require("code-practice").setup()` to override defaults.
Options are merged with |vim.tbl_deep_extend()|.

>lua
    require("code-practice").setup({
      storage = {
        home = vim.fn.stdpath("data") .. "/code-practice",
        db_path = vim.fn.stdpath("data") .. "/code-practice/exercises.db",
        exercises_json = nil,   -- auto-import this file if DB is empty
      },

      ui = {
        width = 0.8,            -- fraction of editor width
        height = 0.8,           -- fraction of editor height
        border = "rounded",
        show_numbers = true,    -- show exercise index in browser list
      },

      -- Engine defaults are built from the registry (see |code-practice-engines|).
      -- Override any engine by name:
      engines = {
        python = { enabled = true, cmd = "python3" },
        rust   = { enabled = false },
      },

      runner = {
        timeout = 5,            -- seconds per test case
        show_time = true,
        auto_save = true,
      },

      ai_hints = {
        enabled = false,        -- set true to use AI hints instead of static
        model = "Qwen/Qwen3-Coder-Next",   -- HF Inference API model
        hf_token_env = "HF_TOKEN",          -- env var holding the HF token
      },

      keymaps = {
        browser = {
          open_item = "<CR>",
          filter_easy = "e",
          filter_medium = "m",
          filter_hard = "h",
          filter_all = "a",
          close = "q",
        },
        exercise = {
          run_tests = "<C-t>",
          show_hint = "<C-i>",
          view_solution = "<C-l>",
          show_description = "<C-d>",
          next_exercise = "<C-n>",
          prev_exercise = "<C-p>",
          skip_exercise = "<C-k>",
          open_browser = "<C-b>",
        },
      },
    })
<

7. ENGINES                                          *code-practice-engines*

Code Practice uses an engine registry (`lua/code-practice/engines.lua`) as
the single source of truth for per-engine metadata. Every other module
reads from this registry instead of hard-coding language specifics.

Each engine entry defines: file type, extension, icon, browser filter key
and label, health-check command, default config, and runner capabilities.

BUILT-IN ENGINES ~

    Engine   Type    Ext   Filter  Runner ~
    python   coding  .py   p       interpreted (wrap_test + run_cmd)
    rust     coding  .rs   r       compiled (wrap_test + compile_cmd + run_cmd)
    theory   theory  .md   t       answer checker

RUNNER DISPATCH ~

The runner selects a path based on engine capabilities, not names:

- `compile_cmd` present  ->  compile-then-run (e.g. rustc)
- `wrap_test` present    ->  interpreted run (e.g. python3)
- `type == "theory"`     ->  multiple-choice checker

ADDING A NEW ENGINE ~

1. Add a registry entry in `lua/code-practice/engines.lua` with the
   required fields (see existing entries for reference).
2. For exercise generation, add a matching section in `tools/engines.toml`.
3. No other files need to change.

8. EXERCISE GENERATION                             *code-practice-generate*

Generate exercises from a syllabus using Hugging Face Inference API models.
The generator is engine-agnostic: the LLM produces a self-contained test
harness alongside each exercise.

REQUIREMENTS ~

- uv (https://github.com/astral-sh/uv)
- HF_TOKEN environment variable (or `huggingface-cli login`)

Run |:checkhealth| |code-practice| to verify both are available.

FROM NEOVIM ~

Use |:CP| `generate`. It will prompt for topic, count, difficulty, and engine.

FROM THE COMMAND LINE ~

>bash
    cd ~/.local/share/nvim/lazy/code-practice
    export HF_TOKEN=your_token

    # Generate from syllabus
    uv run tools/generate_exercises.py tools/syllabus.toml

    # Custom model
    uv run tools/generate_exercises.py tools/syllabus.toml \
        --model Qwen/Qwen3-Coder-30B-A3B-Instruct

    # Dry run (print JSON, don't insert into DB)
    uv run tools/generate_exercises.py tools/syllabus.toml --dry-run

    # Custom engines config
    uv run tools/generate_exercises.py tools/syllabus.toml \
        --engines my_engines.toml
<
CONFIGURATION FILES ~

- `tools/engines.toml`  Engine definitions (run commands, prompt rules,
                        required fields). Add a new engine here.
- `tools/syllabus.toml` What to generate (topics, counts, difficulties).

9. DATA AND STORAGE                                    *code-practice-data*

Exercises are stored in an SQLite database at the path configured by
`storage.db_path` (default: `stdpath("data")/code-practice/exercises.db`).

The database schema is defined in `schema.sql` at the repository root --
the single source of truth shared by the Lua plugin, the test seeder, and
the exercise generator. It contains four tables: exercises, test_cases,
attempts, and theory_options. Foreign keys are enabled; deleting an
exercise cascades to its test cases, attempts, and options.

IMPORTING ~

Use |:CP| `import` to load exercises from a JSON file: >vim
    :CP import /path/to/exercises.json
<
Use the bang form to wipe all existing data and replace: >vim
    :CP! import /path/to/exercises.json
<
Set `storage.exercises_json` in your config to auto-import on first run
when the database is empty.

The expected JSON schema matches `test/example_exercises.json` in the
plugin source.

10. HEALTH CHECK                                     *code-practice-health*

Run `:checkhealth code-practice` to verify:

- Neovim version (0.10+ required)
- nui.nvim and sqlite.lua plugins
- Engine executables based on enabled config (iterates the registry)
- uv executable (needed for |:CP| `generate`)
- curl executable (needed when `ai_hints.enabled` is true)
- HF_TOKEN environment variable (needed for |:CP| `generate` and AI hints)
- Database file existence
